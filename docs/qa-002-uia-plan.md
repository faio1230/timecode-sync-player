# QA-002 対応プラン: 再生中の UIA ツリー取得不能（v0.3 向け）

作成: 2026-07-17。QA レポート QA-002（C:\Users\codea\Documents\qa-report-v0.2.0.md）。
**実行ルールは docs/test-coverage-roadmap.md の「一括実行ルール」に従う。** 上書き事項:
- ブランチ: fix/qa-002-uia。push しない。E2E ゲートは全件（実機込み・Skip なし）
- **U1（診断）が完了して証拠が揃うまで U2（修正）に着手しない**（根本原因なき修正の禁止）

## 症状（確定情報）

- 動画再生中、ウィンドウ全体の UIA ツリー列挙が null / タイムアウトになる
- 個別要素のクエリ（FlaUI の FindFirst 系）は成功する（既存 E2E が全件動作している事実）
- 停止中・再生終了後は全ツリー取得が復帰する
- 仮説: 25fps の WriteableBitmap 更新で UI スレッドが飽和し、UI スレッド上で動く
  WPF UIA プロバイダの全ツリー列挙（大量の同期呼び出し）が処理されない/間に合わない

## U1: 診断（証拠を先に取る）

計測ハーネス（テストプロジェクト内の診断用テストまたはスクリプトで可・コミットは任意）で
再生中に以下を計測し、結果を数値で progress に記録する:
1. UIA 全ツリー列挙の所要時間/失敗率（停止中 vs 再生中、要素数も記録）
2. UI スレッドの Dispatcher 遅延（再生中に `DispatcherPriority.Input` 等へ投げたコールバックの
   遅延分布）— UI スレッド飽和仮説の直接検証
3. レンダー更新の実頻度（mpv render callback → WriteableBitmap 更新の毎秒回数と1回あたり時間、
   動画 fps との関係）
4. 対照実験: 表示更新を人為的に 1/2（例: 2回に1回スキップ）へ落とした場合に UIA が復帰するか
   （一時的な実験変更。コミットしない）
結論として「どの処理が UI スレッドの何%を占め、UIA のどの呼び出しが詰まるのか」を特定する。

## U2: 修正（U1 の証拠に基づき選択。候補と制約）

候補（U1 の結果で選ぶ。複数併用可）:
- **表示スロットリング**: 表示用の WriteableBitmap 更新を上限 fps（例: 動画 fps と等値まで、
  余剰コールバックをスキップ）に制限する。`RenderUpdateScheduler` に判定を実装し
  ユニットテスト可能な純ロジックにする
- **Dispatcher 優先度調整**: レンダー更新の投げ先優先度を Input/UIA 処理より下げる
- **AddDirtyRect の範囲最適化**（全面 → 実変更域）が計測上有効なら
制約:
- **同期精度・映像の見た目（テアリング・カクつき）を劣化させない** — 実機 LTC E2E 全件 +
  劣化信号シナリオが無変更で通ること。表示スロットリングを入れる場合も Spout への送信頻度は
  現行を維持する（Spout 出力は表示と独立に全フレーム送る — 受け側ツールの品質に直結）
- CLAUDE.md の既知問題（デリゲート保持・構造体レイアウト・P/Invoke）は不変
- 挙動判定ロジック（同期・ギャップ・信号断）に触らない

## U3: 回帰テストと文書

1. **QA-002 の再現を E2E 化**: 再生中にウィンドウの UIA 全ツリー列挙（相当の広域列挙）が
   規定時間内に成功することをアサートする（修正前に赤・修正後に緑の RED→GREEN を記録）
2. スロットリング等の判定ロジックのユニットテスト + ミューテーション確認
3. 実機 LTC 全件 + 全 E2E で同期・表示の無劣化を確認
4. CHANGELOG の Known limitations から該当項目を削除（修正できた範囲を正確に記載）。
   部分的改善に留まる場合は制限の記述を実態に更新
